# Auditoría técnica — TravelApp
Fecha: 2026-04-29
Ejecutada por: Claude Code

---

## Resumen ejecutivo

El proyecto tiene una base técnica sólida con ~15.000 líneas de código TypeScript distribuidas en 110 archivos. Los módulos críticos del MVP (auth, viajes, itinerario, gastos, documentos, perfil y explorar) están implementados y funcionales. La suite de tests pasa al 100% (352 tests, 23 archivos). Existen tres problemas bloqueantes: el typecheck falla por incompatibilidad entre TypeScript 5.3 y Expo 54, el `CLAUDE.md` documenta versiones del stack que no coinciden con las instaladas (SDK 52 vs 54 real), y `packages/types/database.ts` no ha sido generado por el CLI de Supabase. Hay también una pantalla stub (`documents.tsx`) y dos archivos de ruta que faltan (`callback.tsx`, `welcome/index.tsx`).

**Causa raíz**: Expo 54 incluye `"module": "preserve"` en su `tsconfig.base.json`, opción que requiere TypeScript ≥ 5.4. El proyecto tiene `typescript: ~5.3.3` en `package.json`, que no soporta esta opción.

**Consecuencia**: `pnpm typecheck` siempre falla, lo que significa que errores de tipos en el código del proyecto no se detectan en CI. El checklist de CLAUDE.md ("pnpm typecheck pasa con 0 errores") no se puede cumplir actualmente.

**Solución**: Actualizar TypeScript a `^5.4.0` o superior en `apps/mobile/package.json`.

## 8. Riesgos técnicos identificados

### CRÍTICO

**R1 — TypeScript 5.3 incompatible con Expo 54**
- `pnpm typecheck` siempre falla con error en `expo/tsconfig.base.json`
- Ningún error de tipos en el código del proyecto es detectable automáticamente
- **Acción**: `typescript: "~5.4.0"` o `"^5.5.0"` en `package.json`

**R2 — packages/types/database.ts no existe**
- Hay un comentario en `useTrips.ts` que menciona este archivo como pendiente
- Sin los tipos generados, el auto-complete y typecheck de queries Supabase no funciona
- **Acción**: Con Supabase local corriendo: `supabase gen types typescript --local > packages/types/database.ts`

**R3 — get-explore-feed importa desde ../../packages/types con ruta relativa**
- `import type { ... } from '../../packages/types/explore.ts'`
- Esta ruta sale del directorio `supabase/functions/` y accede a `packages/`
- En Deno Deploy, cada Edge Function se despliega de forma aislada y puede no tener acceso a ese path
- **Acción**: Inline los tipos directamente en la Edge Function (duplicación intencional, como se hace en `apps/mobile/constants/destinations.ts`)

### ALTO

**R4 — CLAUDE.md desactualizado**
- Documenta Expo 52 / React 18 / Router v4 pero el proyecto usa Expo 54 / React 19 / Router v6
- Claude generará código basado en APIs/versiones incorrectas en futuras sesiones
- **Acción**: Actualizar CLAUDE.md con las versiones reales

**R5 — delete-account usa esm.sh en lugar de npm:**
- Todas las demás Edge Functions usan `import { createClient } from 'npm:@supabase/supabase-js@2'`
- delete-account usa `import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'`
- esm.sh puede tener latencia adicional en cold starts; npm: está cacheado por Deno
- **Acción**: Cambiar a `npm:@supabase/supabase-js@2`

**R6 — Tab Documentos es un stub**
- `(tabs)/documents.tsx` solo muestra "Próximamente"
- La pantalla real de documentos existe en `trips/[id]/documents/index.tsx` pero solo para documentos de un viaje específico
- Un tab de documentos globales (sin trip_id) puede ser parte del roadmap pero no está implementado
- **Acción**: Implementar o cambiar el tab a redirigir al viaje activo

### MEDIO

**R7 — console.log en Edge Functions parse-document y edit-node**
- Viola la regla PROHIBIDO de CLAUDE.md
- En Deno/Supabase Edge Functions es técnicamente aceptable (van a los logs del servidor)
- Pero dificulta mantener consistencia en el estilo del proyecto
- **Acción**: Opcional; documentar en CLAUDE.md que Edge Functions pueden usar console.log para logging de servidor

**R8 — logger.ts y Sentry son stubs**
- En producción, errores del cliente no llegarán a Sentry
- `EXPO_PUBLIC_SENTRY_DSN` está en .env.local.example pero `@sentry/react-native` no está instalado
- **Acción**: Fase 9 (según CLAUDE.md). Documentar que hasta entonces los errores solo se ven en desarrollo.

**R9 — Rutas faltantes**
- `app/(auth)/callback.tsx` no existe. El handler de Magic Link está en `_layout.tsx` via `Linking.addEventListener`. Si el deep link llega cuando la app no está en memoria, puede no procesarse correctamente en todos los platforms.
- `app/(welcome)/index.tsx` no existe. Si algo navega a `/(welcome)` (sin el nombre de pantalla específico), Expo Router lanzará un error 404.

**R10 — ScreenSkeleton y supabaseInterceptor referenciados pero no existen**
- `ScreenSkeleton.tsx` aparece en el audit spec como componente esperado pero no existe
- `supabaseInterceptor.ts` idem. No están importados en ningún archivo activo, por lo que no causa errores en runtime actualmente.
